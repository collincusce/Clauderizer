## Project Lessons (most relevant to this phase)

_(5 of 25 shown — ranked by keyword + entity-id overlap with this phase, no ML. The full set is canonical in `docs/LESSONS.md`; the handoff focuses under memory pressure without dropping anything from canonical memory.)_

**L-07.** Design failure reporting backward from the channel the consumer actually reads, then place the reliability wrapper on the deepest layer whose failure still leaves that channel reachable. The harness injects only a hook's stdout, so a dying hook's informative stderr is indistinguishable from silence — a breadcrumb wrapper is channel REROUTING (capture 2>&1, reprint on stdout, exit 0) as much as failure detection. And the wrapper's layer is a coverage decision, not a host-affinity default: enumerate which failure modes each candidate layer can observe AND report through the consumer-read channel (a Windows-side cmd wrapper would add UNC noise to healthy sessions while only covering repo-unreachable states). H-08 closed the loop: there is always a layer below your deepest wrapper (the harness's own shell), so only in-band evidence — the digest arriving — proves the chain. *(from 2026-06-09-agent-autonomy)*
**L-39.** A deterministic relevance/gain kill-gate concludes only what its fixtures permit. When the eval corpus is cleanly separable (disjoint topics — as tests/benchmarks RANKER_ENTRIES is), the baseline saturates at the 1.0 recall/nDCG/MRR ceiling, so NO scoring change can show lift: the gate proves "no regression", never "no value", and can only return DISCARD/break-even by construction. To FALSIFY "richer scorer X helps", build the adversarial fixture targeting X's mechanism FIRST — for BM25/Okapi: length-bias confounds, term-frequency skew, near-duplicate topics. Extends L-32 (saturated target ⇒ predict the null) with a fixture-design corollary. Measured instance (2026-06-24-ranker-spike): stdlib BM25 (k1=1.5, b=0.75) tied the raw-overlap baseline at 1.0000 on every extraction+multi_session metric and across a full b-sweep, with guards flat (multi_session precision 0.75, knowledge_updates contradiction 0.0) ⇒ a clean measured DISCARD. Watch-out for any future ranker swap: BM25 length-normalization breaks the stale-vs-current SECONDARY-sort tie that supersession-demotion (analyze.rank_relevant) relies on, so it needs a length-adversarial knowledge_updates fixture before it can be called safe. *(from 2026-06-24-ranker-spike)*
**L-31.** Before any irreversible release step (tag → GitHub Release → PyPI publish), gate on two sweeps. (1) A version number is a claim across four registries that never sync — source (pyproject/__version__), remote git tags, GitHub Releases, the PyPI index — plus the ordering invariant that origin/main must hold the staged commit BEFORE any tag/Release (a GitHub-UI release tags the REMOTE head, so locally-authored guards are unpushed by construction); sweep all four with fresh eyes (uvx by-name answers from uv's cache and can hide a failed attempt — v0.7.0/v0.8.0 were double-claimed by exactly this). (2) Run the suite on EVERY host leg the CI matrix covers — a green on one OS is a guess about the others and the publish cannot be undone; a path/separator assertion is itself a platform claim (assert the wrapper FILE hook.sh|hook.cmd, not the slash). 0.14.0 shipped with 3 Windows cells red on such an assertion. (Consolidates L-08, L-20.) *(from 2026-06-21-cross-host-cross-model-clauderizer-universal-agents-md-mcp-substrate)*
**L-16.** Generated/managed content has a source template — edit the source, not just the render. Clauderizer renders CLAUDE.md's stanza and .claude/skills from src/clauderizer/templates/claude_stanza.md and src/clauderizer/skills/ at init time; editing only the rendered copy leaves the source stale (and a future init would overwrite the render), while editing only the source leaves the live repo stale until re-init. No test enforces sync, so the drift is silent — update both, source first. *(from 2026-06-17-spec-kit-discipline-gates)*
**L-40.** When you author BOTH an eval fixture and the detector it scores, a 100%-detection / 0-false-positive result is suspicious by construction — you may be teaching to the test (the confirmation-bias face of L-32/L-39). Give the fixture discriminating power: seed adversarial NEAR-MISSES (a long-but-hollow note; a paper citation that is ALSO in-repo-anchored; a factual "all criteria met" with no intensifier) and run a deliberately NAIVE strawman detector beside the real one. The result is credible only if the principled detector beats the strawman ON the near-misses (a real mechanism), not merely beats a prior no-check baseline (a tautology — anything beats zero). Extends L-39: don't just build the adversarial fixture first — build a strawman the fixture must DEFEAT. Measured: 2026-06-24-critique-bias-hardening, principled 100% det / 0% FP vs naive 20-50% FP; naive wrong on 6/7 near-misses, principled on 0. *(from 2026-06-24-critique-bias-hardening)*
