## Phase Notes (agent-owned)

**What Phase 0 left you.** `src/clauderizer/locking.py` exposes
`write_lock(lock_path, *, acquire_timeout=None, stale_timeout=None)`; defaults
live in module attributes (`DEFAULT_ACQUIRE_TIMEOUT` 10s, `DEFAULT_STALE_TIMEOUT`
30s) resolved at call time — tests monkeypatch them. All 18 public
`mutations.*` functions are wrapped by `mutations._locked`, which acquires
`paths.write_lock_file` (`.clauderizer/write.lock`) around the whole
read-modify-write. The lock is reentrant per thread, so ops can freely call
mutations that call mutations.

**Task 1.2 gets locking for free — except for the residue.** Ops that execute
`mutations.*` already serialize. But two write surfaces do NOT acquire the lock
yet (recorded as deferred in `_cascade-reports/2026-06-09-subsys.mutations-01.md`):
`cascade.run` when invoked directly by the `cz_cascade` tool, and the ritual
writes (preflight baseline refresh, handoff regeneration). The shared ops
registry is the natural single place to wrap **every** write op with
`locking.write_lock(paths.write_lock_file)` uniformly — do that rather than
chasing call sites.

**Registry source of truth.** The only existing name→callable map is the
`@mcp.tool()` definitions in `src/clauderizer/mcp_server.py` (24 tools);
`src/clauderizer/tools_list.py` also enumerates the tool surface — check it
before inventing a third list. Task 1.3's parity test should enumerate from
one shared table so MCP dispatch and `clauderize ops` cannot drift.
