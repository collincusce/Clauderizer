### H-05 — No write lock on tracked docs: concurrent sessions on one repo can silently lose updates or duplicate IDs

- **Severity**: medium
- **Status**: resolved (2026-06-09)
- **Affected**: mutations.py (all 18 public write functions); .clauderizer index cache; any multi-window or multi-agent workflow on one repo
- **Invariant violated**: Append-only trackers assume appends are serialized; numbering assumes allocation+write is atomic
- **Preconditions**: Two writer processes (MCP servers, future CLI ops, or shims) mutating the same repo concurrently; sub-second collision window per write
- **Impact**: Every mutation is read-modify-write on markdown with no inter-process lock (verified: no flock/lockfile/mutex anywhere in src/clauderizer; the only 'lock' is profile.lock.toml, a pin file). Two sessions - or one session plus a subagent - mutating the same repo in the same window can last-writer-wins erase each other's appends, and two concurrent ID allocations (H-NN, L-NN, D-k, A-NNN) can both compute the same next number. Cross-project concurrency is unaffected: state is per-repo by construction (server per session, repo = spawn cwd).
- **Root cause**: The engine assumes one interactive session per repo, but stdio MCP spawns one server process per session - the assumption breaks the moment a second window or subagent opens the same project
- **Reproduction**: Open two sessions on one repo and fire cz_add_lesson in both within the same second; inspect numbering and the lessons section for a lost append or duplicate number
- **Recommended fix**: Advisory lock in the single mutation path (D-007 makes it one choke point): e.g. .clauderizer/write.lock created O_EXCL with a stale-lock timeout, held per mutation. Covers MCP, CLI ops, and shims uniformly; read tools stay lock-free (L-03). Until then: one writer session per repo at a time; reads are always safe; git history is the recovery backstop. Candidate task for the agent-autonomy gameplan Phase 0 alongside clauderize ops
- **Regression tests**: None yet - candidate: N concurrent add_lesson processes against one repo yield N distinct sequential numbers and N surviving entries
- **Resolution**: Closed by agent-autonomy Phase 0: advisory write lock at the mutation choke point (src/clauderizer/locking.py; all 18 public mutations.* functions serialize on .clauderizer/write.lock, reentrant, stale takeover ~30s). Regression evidence: tests/test_locking.py::test_concurrent_writer_processes_lose_nothing (8 concurrent writer processes -> 8 distinct sequential ids, 8 surviving appends) and test_crashed_holder_blocks_at_most_stale_timeout. This resolution was itself written through the locked path.

### H-09 — Hook digest silently empty when executor cwd is not the repo: engine discovers the repo from cwd, wrapper does not anchor, cmd.exe cannot hold a UNC cwd

- **Severity**: medium
- **Status**: resolved (2026-06-10)
- **Affected**: clauderizer-hook repo discovery (cwd walk); the generated .clauderizer/hook.sh (inherits cwd instead of anchoring to the repo it was generated for); init's spawn-test and doctor's hook probe (neither varies cwd, so neither can catch this).
- **Invariant violated**: H-01 / L-07: the silence channel reopened below the wrapper's observable surface — the wrapper ran fine, the engine answered "no repo here" with empty/0, and that is indistinguishable in-band from a healthy hook in a repo-less directory (L-09: only the digest arriving proves the chain).
- **Preconditions**: Executor or harness whose spawn cwd differs from the project repo: cmd.exe with any UNC-rooted project (structural), or any client that spawns hooks from a fixed directory.
- **Impact**: Any harness or executor that spawns the SessionStart command without preserving the project cwd gets an empty exit-0 hook — no digest AND no breadcrumb (the wrapper runs, but the engine prints nothing and the wrapper's [ -n "$out" ] branch stays silent). Found live in Phase 0: the matrix's entire cmd.exe column was empty because cmd cannot hold the UNC project cwd (\\wsl.localhost\...) and falls back to C:\Windows; reproduces equally from PowerShell with cwd C:\ and from the engine directly with --cd /tmp. A future harness spawning hooks from the user home would silently kill the digest on every host.
- **Root cause**: The engine hook resolves its repo from the inherited working directory, and the generated wrapper — which is per-repo and knows its repo path statically (its own engine-hook path is already hardcoded) — does not cd to that repo before delegating.
- **Reproduction**: wsl.exe -d ubuntu --cd /tmp /home/ccusce/Clauderizer/.venv/bin/clauderizer-hook → 0 bytes, exit 0. cmd /d /c "wsl.exe -d ubuntu /bin/sh /home/ccusce/Clauderizer/.clauderizer/hook.sh" with UNC cwd → 0 bytes on BOTH streams (file-redirect verified), exit 0. Control: an anchored variant (cd /home/ccusce/Clauderizer before delegating; /tmp/hook-anchored.sh) emits the full 758-byte digest from cwd C:\ under all three executors.
- **Recommended fix**: Phase 1: hosts.py generates a self-anchoring wrapper (cd <repo> || exit 0 before invoking the engine hook — keep exit 0 so a deleted repo stays a silent no-op for the harness, or emit a breadcrumb naming the missing repo). Phase 2: init's spawn-test and doctor's hook probe must spawn from a NON-repo cwd so anchoring is what they verify. scripts/wiring_matrix.ps1 already runs hostile-cwd (C:\) by default.

### H-10 — Windows: a finished writer orphans its own lock when unlink races a concurrent reader

- **Severity**: medium
- **Status**: resolved (2026-06-20)
- **Affected**: src/clauderizer/locking.py (_release_file); tests/test_locking.py::test_concurrent_writer_processes_lose_nothing
- **Invariant violated**: Extends H-05: the advisory write lock must not spuriously deny a writer when no live holder exists. No lost-update/duplicate-id occurs (takeover recovers), so this is a liveness/robustness gap, not a correctness violation.
- **Preconditions**: Windows host; two or more processes contending for one repo's .clauderizer/write.lock concurrently (waiters polling read_holder while a holder releases). POSIX is unaffected (unlink succeeds on open files).
- **Impact**: On Windows, os.unlink in locking._release_file raises a sharing violation (PermissionError) when another process has the lock file open. Waiters poll read_holder, holding the file open for an instant on each poll, so a just-finished writer's release can fail; the error was swallowed (except OSError: pass), orphaning the lock. The orphan is then recovered only by stale takeover (~stale_timeout, default 30s), so a second writer racing the same repo's write.lock gets a spurious, retryable LockHeld for up to ~30s even though the first writer already completed. Surfaced as intermittent failures of tests/test_locking.py::test_concurrent_writer_processes_lose_nothing on windows-latest CI cells (seen on py3.11 and py3.12 on different runs), which red-flagged the 0.14.1 release matrix.
- **Root cause**: _release_file caught and swallowed the unlink OSError instead of retrying. Windows refuses to delete a file with an open handle, and concurrent pollers keep one open intermittently, so the holder's single unlink attempt can lose the race and leave the file behind.
- **Reproduction**: Run tests/test_locking.py::test_concurrent_writer_processes_lose_nothing (8 writer processes) on a slow/contended windows-latest runner: intermittently a waiter raises LockHeld 'waited Ns' while the holder field shows a since-timestamp older than the wait -- an orphaned lock not recovered within the acquire budget. Originally masked by acquire_timeout (10s) < stale_timeout (30s), so takeover never fired before the waiter gave up.
- **Recommended fix**: _release_file should retry the unlink for a short window (with the nonce guard re-checked each pass) before falling back to stale takeover. Deterministic regression test that simulates a transient unlink failure. (Defense in depth: the concurrent-writer test also widens the child acquire budget far above the stale window so takeover can rescue a waiter if an orphan ever slips through.)
- **Regression tests**: tests/test_locking.py::test_release_retries_unlink_past_transient_oserror (monkeypatched unlink fails twice then succeeds; asserts retry and no orphan).
- **Resolution**: Fixed in PR #15 (branch fix/windows-lock-orphan-release-retry): locking._release_file now retries the unlink (~2s, _RELEASE_UNLINK_ATTEMPTS, with the nonce guard re-checked each pass) before falling back to stale takeover, so a finished writer no longer orphans its lock when os.unlink loses a race with a concurrent reader on Windows. POSIX returns on the first attempt. Regression test tests/test_locking.py::test_release_retries_unlink_past_transient_oserror added (monkeypatched unlink fails twice then succeeds; asserts retry + no orphan); the concurrent-writer test retains the widened acquire budget from PR #14 as a backstop. Verified: full suite green locally; CI matrix green across 3 consecutive runs (all windows cells, py3.11/3.12/3.13).
