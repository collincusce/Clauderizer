# Trust Model

> What Clauderizer writes into your repo, what executes when, and the
> contracts each surface honors. Every behavioral claim cites the code that
> implements it — if a claim and the code ever disagree, the code is the
> truth and the claim is a bug (file it).

## The one-paragraph version

`clauderize init` writes files into **your repo only** — never your home
directory, shell profile, or system. Two of those files register commands
that your agent harness (e.g. Claude Code) executes: an MCP server
registration and SessionStart + UserPromptSubmit hooks (the same wrapper
command, registered under two events). Clauderizer itself never installs a
daemon, never phones home, has **zero runtime dependencies**, and contains
no self-executing path: everything it registers runs only when *your*
harness, under *your* trust settings, chooses to run it.

## What `init` writes (the complete list)

All writes happen in `clauderizer.scaffold.init.init()` — read it; it's one
function.

| Surface | What | Contract |
|---|---|---|
| `.mcp.json` | a `clauderizer` entry under `mcpServers` | key-merge only: other servers are preserved verbatim (`_register_mcp`) |
| `.claude/settings.json` | SessionStart + UserPromptSubmit hook entries (the same wrapper command) | replaces only entries matching `hosts.is_hook_command`; unrelated hooks preserved (`_register_hook`, `HOOK_EVENTS`) |
| `.clauderizer/` | `config.toml`, `profile.lock.toml`, the hook wrapper (`hook.sh`/`hook.cmd`), `kimi-setup.md` (a non-destructive kimi-code wiring guide), a gitignored `index.json` cache | engine-owned; regenerated by `init`; `profile.lock.toml` is written once and then preserved (your edits survive) |
| `CLAUDE.md` | a stanza between `<!-- clauderizer:start -->` / `<!-- clauderizer:end -->` markers | only the marker block is ever rewritten (`writer.upsert_marker_block`); your text outside it is byte-preserved |
| `AGENTS.md` | the same Clauderizer stanza, for AGENTS.md-aware hosts (kimi-code, Codex) | identical marker-block contract; one source stanza, so it cannot drift from `CLAUDE.md` (`init.py` step 8b) |
| `.claude/skills/clauderizer-*` | the six workflow skills | engine-owned; refreshed on re-init |
| `docs/` | scaffold templates (VISION, DECISIONS, …) | created only if absent (`writer.create_if_absent`) — never clobbered, and **yours**: uninstalling keeps them ([UPGRADING.md](UPGRADING.md)) |

Before writing anything, `init` **spawn-tests** every command it is about to
register and refuses to write wiring that does not launch and identify
itself (`WiringRefused`; the H-04 guard). It also refuses to register paths
inside uv's ephemeral cache, which would die on `uv cache clean`
(`_under_uv_cache`).

## What executes, and when

**The SessionStart hook** — your harness executes the registered command
string at session start. The command runs a thin wrapper
(`hosts.render_hook_wrapper`) whose contract is:

- `cd` to the repo it was generated for (never inherits a surprise cwd);
- run the engine hook and print the status digest to **stdout** (the only
  channel the harness injects into context);
- convert any engine failure into a one-line breadcrumb on stdout instead
  of silence;
- **always exit 0** — a memory tool must never block or fail a session.

The engine hook (`clauderizer.hook.dispatch`) reads your repo's
`docs/` and `.clauderizer/` and prints text. It writes nothing.

**The UserPromptSubmit hook** — the same wrapper command, registered under a
second event (`init.py` `HOOK_EVENTS`). On each prompt the dispatcher runs the
read-only analyze gate and surfaces the recorded decisions/invariants most
relevant to what you typed — a pointer into canonical memory, silent when
nothing is relevant. Like every hook handler it only reads and always exits 0
(INVARIANT-06). (`PreCompact`/`PostCompact` handlers also exist for kimi-code,
which injects their stdout; Claude Code drops it, so `init` does not register
them there.)

**The MCP server** (`clauderizer-mcp`) is spawned by your MCP client per
`.mcp.json`, speaks stdio, and operates on the repo it runs in. Its tools
are the `cz_*` read/write surface — reads are side-effect-free by contract
(L-03: a context fetch must never mutate), writes serialize on a repo-local
lock file (`.clauderizer/write.lock`).

**Nothing else executes.** No daemon, no scheduler, no network calls at
runtime (the engine's only network user is `clauderize release-check`,
a maintainer command that queries PyPI/GitHub read-only when *you* run it).

## Cloning a repo that already carries Clauderizer wiring

The wiring files are repo content, so a cloned repo arrives with them. What
that means on your machine:

- **Nothing runs on clone.** Execution happens only when you open an agent
  session there, and your harness's own trust flow (e.g. Claude Code's
  folder-trust prompt) gates hooks and MCP servers for new directories.
- The registered commands resolve on *your* machine: a venv path from the
  author's machine simply fails — visibly, as an
  `[Clauderizer] engine unreachable` breadcrumb, never silently — and the
  durable `uvx -q --from clauderizer …` form runs the published PyPI
  package under your uv installation.
- Review the wiring like any other repo content: `.mcp.json`,
  `.claude/settings.json`, and `.clauderizer/hook.sh` are short, plain
  text, and written to be read.

## Supply chain

- **Zero runtime dependencies** (`pyproject.toml`: `dependencies = []`;
  stdlib only). The MCP SDK is an explicit opt-in extra
  (`clauderizer[mcp]`).
- Releases publish via **PyPI Trusted Publishing (OIDC)** from GitHub
  Actions — no long-lived tokens exist
  (`.github/workflows/publish.yml`).
- The publish workflow **refuses tag/source skew** (the release tag must
  match the tagged tree's declared version) before building anything.
- `clauderize release-check` gates every release on a fresh sweep of all
  four version registries (local tags, remote tags, GitHub Releases, the
  PyPI index directly) plus push ordering — see
  [RELEASING.md](RELEASING.md).
- The quickstart is executed against the **published package** on a clean
  CI runner on every push and weekly
  (`.github/workflows/quickstart.yml`).
