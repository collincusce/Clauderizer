# Trust Model

> What Clauderizer writes into your repo, what executes when, and the
> contracts each surface honors. Every behavioral claim cites the code that
> implements it — if a claim and the code ever disagree, the code is the
> truth and the claim is a bug (file it).

## The one-paragraph version

`clauderize init` writes files into **your repo only** — never your home
directory, shell profile, or system. Two of those files register commands
that your agent harness (e.g. Claude Code) executes: an MCP server
registration and SessionStart + UserPromptSubmit hooks (the same wrapper
command, registered under two events). Clauderizer itself never installs a
daemon, never phones home, has **zero runtime dependencies**, and contains
no self-executing path: everything it registers runs only when *your*
harness, under *your* trust settings, chooses to run it.

## What `init` writes (the complete list)

All writes happen in `clauderizer.scaffold.init.init()` — read it; it's one
function.

Bare `init` wires **all** project-level hosts by default (`[host] enabled = ["*"]`,
D-046). `--host <name>` scopes a run to one host. Session preference is still
`[host] target` (display / doctor primary); runtime bootstrap uses env detection
(D-047). The MCP-config surface and hook surface are written per host; everything
else is the same across hosts.

| Surface | What | Contract |
|---|---|---|
| host MCP config | a `clauderizer` entry under that host's server key | key-merge only: other servers are preserved verbatim (`_register_mcp` for claude-code, `hosttargets.emit_mcp` for other auto-write hosts). Path/key per host (see the per-host table below). |
| `.claude/settings.json` | SessionStart + UserPromptSubmit hook entries (the same wrapper command) — **only for the `claude-code` host target** (its native hook file) | replaces only entries matching `hosts.is_hook_command`; unrelated hooks preserved (`_register_hook`, `HOOK_EVENTS`). Other hook-capable hosts get a `.clauderizer/<host>-hook-setup.md` guide instead (their hook schemas are unverified, so guided not auto-written) |
| native-instructions file | `.continue/rules/clauderizer.md` (Continue), `GEMINI.md` (Gemini CLI) — **only for hosts that do not read `AGENTS.md`** | the instructions floor (call `cz_status` first); marker-block contract; every other supported host reads `AGENTS.md`, which already carries the floor |
| `.clauderizer/` | `config.toml` (size dial + host profile + **`[host] target`** + session host + active gameplan), `profile.lock.toml`, the hook wrapper (`hook.sh`/`hook.cmd`), per-host setup guides (`<host>-mcp-setup.md` for guide-only MCP hosts, `<host>-hook-setup.md` for hook-capable hosts, `kimi-setup.md`), a gitignored `index.json` cache | engine-owned; regenerated by `init`; `profile.lock.toml` is written once and then preserved (your edits survive) |
| `CLAUDE.md` | a stanza between `<!-- clauderizer:start -->` / `<!-- clauderizer:end -->` markers | only the marker block is ever rewritten (`writer.upsert_marker_block`); your text outside it is byte-preserved |
| `AGENTS.md` | the same Clauderizer stanza, for AGENTS.md-aware hosts (most supported hosts) | identical marker-block contract; one source stanza, so it cannot drift from `CLAUDE.md` (`init.py` step 8b) |
| `.claude/skills/clauderizer-*` | the six workflow skills | engine-owned; refreshed on re-init |
| `docs/` | scaffold templates (VISION, DECISIONS, …) | created only if absent (`writer.create_if_absent`) — never clobbered, and **yours**: uninstalling keeps them ([UPGRADING.md](UPGRADING.md)) |

### Per-host MCP config — where the entry lands

For every **non-`claude-code`** host the emitted command is machine-INDEPENDENT
(`uvx --from "clauderizer[mcp]" clauderizer-mcp`; absolute venv paths and `wsl.exe`
shims are refused at write time for multi-host / auto-write emitters —
`hosttargets.is_path_safe`), because those configs are the ones most likely to be
committed and shared. Scoped Claude-only init may still use a local/shimmed
`.mcp.json` for dogfood reliability (gitignored when not portable). The
`claude-code` `.mcp.json` is composed by `_resolve_invocation` and prefers your
LOCAL install (a venv/console-script path, or a `wsl.exe` shim for a
Windows→WSL session host) for launch reliability — so it can be machine-specific
**by design**; it is your local wiring, and the dogfood repo's own `.mcp.json`
is gitignored for exactly that reason. `hosttargets.path_safety_audit`
(run in the test suite, gitignore-aware) scans the committable per-host configs
and flags any machine-specific path that would leak through a commit. Hosts with
a JSON config are written automatically (a non-destructive merge that preserves
other servers); hosts whose config is TOML (no stdlib writer) or global get a
setup guide in `.clauderizer/<host>-mcp-setup.md` instead.

| Host | File | Server key | Mode |
|---|---|---|---|
| Claude Code | `.mcp.json` | `mcpServers` | auto-write |
| Cursor | `.cursor/mcp.json` | `mcpServers` | auto-write |
| GitHub Copilot / VS Code | `.vscode/mcp.json` | `servers` | auto-write |
| Continue.dev | `.continue/mcpServers/clauderizer.json` | `mcpServers` | auto-write |
| Zed | `.zed/settings.json` | `context_servers` | auto-write |
| Gemini CLI | `.gemini/settings.json` | `mcpServers` | auto-write |
| Cline | `.cline/mcp.json` | `mcpServers` | auto-write |
| Amp | `.amp/settings.json` | `amp.mcpServers` | auto-write (then run `amp mcp approve clauderizer`) |
| OpenAI Codex CLI | `.codex/config.toml` | — | guide-only (TOML) |
| Windsurf | `~/.codeium/windsurf/mcp_config.json` | — | guide-only (global) |
| kimi (Kimi Code CLI) | `.kimi-code/mcp.json` | `mcpServers` | auto-write (hooks guide-only, TOML) |

Before writing anything, `init` **spawn-tests** every command it is about to
register and refuses to write wiring that does not launch and identify
itself (`WiringRefused`). It also refuses to register paths
inside uv's ephemeral cache, which would die on `uv cache clean`
(`_under_uv_cache`).

## What executes, and when

**The SessionStart hook** — your harness executes the registered command
string at session start. The command runs a thin wrapper
(`hosts.render_hook_wrapper`) whose contract is:

- `cd` to the repo it was generated for (never inherits a surprise cwd);
- run the engine hook and print the status digest to **stdout** (the only
  channel the harness injects into context);
- convert any engine failure into a one-line breadcrumb on stdout instead
  of silence;
- **always exit 0** — a memory tool must never block or fail a session.

The engine hook (`clauderizer.hook.dispatch`) reads your repo's
`docs/` and `.clauderizer/` and prints text. It writes nothing.

**The UserPromptSubmit hook** — the same wrapper command, registered under a
second event (`init.py` `HOOK_EVENTS`). On each prompt the dispatcher runs the
read-only analyze gate and surfaces the recorded decisions/invariants most
relevant to what you typed — a pointer into canonical memory, silent when
nothing is relevant. Like every hook handler it only reads and always exits 0.
(`PreCompact`/`PostCompact` handlers also exist for kimi-code,
which injects their stdout; Claude Code drops it, so `init` does not register
them there.)

**The MCP server** (`clauderizer-mcp`) is spawned by your MCP client per
`.mcp.json`, speaks stdio, and operates on the repo it runs in. Its tools
are the `cz_*` read/write surface — reads are side-effect-free by contract
(a context fetch must never mutate), writes serialize on a repo-local
lock file (`.clauderizer/write.lock`). The skill tools keep that contract:
`cz_discover_skills` only **reads** `SKILL.md` frontmatter (name + description)
from your local skill directories and proposes — it never executes a skill or
writes; `cz_register_skill` / `cz_obsolete_skill` write only to `docs/SKILLS.md`.

**Nothing else executes.** No daemon, no scheduler, no network calls at
runtime (the engine's only network user is `clauderize release-check`,
a maintainer command that queries PyPI/GitHub read-only when *you* run it).

## Removing the wiring

`clauderize uninstall` reverses Clauderizer's footprint
(`clauderizer.scaffold.uninstall`). With **no flag** it removes the full
footprint: the `clauderizer` key from `.mcp.json` and every per-host MCP config,
the clauderizer hooks (and the wrapper) from `.claude/settings.json`, the marker
stanzas in `CLAUDE.md`/`AGENTS.md` and the native-instructions floor files, the
`clauderizer-*` skills, the `.clauderizer/` directory, and the one
`.gitignore` line `init` added. `--host <name>` limits it to that host's
footprint. Every removal is key/marker-scoped — other MCP servers, foreign
hooks, and your own content are preserved verbatim (the mirror of the
`_register_mcp` / `upsert_marker_block` contracts). As with the rest of
Clauderizer, your **`docs/` memory is never touched** — uninstall removes
wiring, not memory, so a later `clauderize init` restores the wiring against the
same durable docs (see [UPGRADING.md](UPGRADING.md)).

## Cloning a repo that already carries Clauderizer wiring

The wiring files are repo content, so a cloned repo arrives with them. What
that means on your machine:

- **Nothing runs on clone.** Execution happens only when you open an agent
  session there, and your harness's own trust flow (e.g. Claude Code's
  folder-trust prompt) gates hooks and MCP servers for new directories.
- The registered commands resolve on *your* machine: a venv path from the
  author's machine simply fails — visibly, as an
  `[Clauderizer] engine unreachable` breadcrumb, never silently — and the
  durable `uvx -q --from clauderizer …` form runs the published PyPI
  package under your uv installation.
- Review the wiring like any other repo content: `.mcp.json`,
  `.claude/settings.json`, and `.clauderizer/hook.sh` are short, plain
  text, and written to be read.

## Supply chain

- **Zero runtime dependencies** (`pyproject.toml`: `dependencies = []`;
  stdlib only). The MCP SDK is an explicit opt-in extra
  (`clauderizer[mcp]`).
- Releases publish via **PyPI Trusted Publishing (OIDC)** from GitHub
  Actions — no long-lived tokens exist
  (`.github/workflows/publish.yml`).
- The publish workflow **refuses tag/source skew** (the release tag must
  match the tagged tree's declared version) before building anything.
- `clauderize release-check` gates every release on a fresh sweep of all
  four version registries (local tags, remote tags, GitHub Releases, the
  PyPI index directly) plus push ordering — see
  [RELEASING.md](RELEASING.md).
- The quickstart is executed against the **published package** on a clean
  CI runner on every push and weekly
  (`.github/workflows/quickstart.yml`).
